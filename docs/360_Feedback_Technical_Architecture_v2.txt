360° Feedback & Performance Intelligence System Technical Architecture Document

1. EXECUTIVE SUMMARY
1.1 Business Context
Problem:
Organizations require structured, secure, and psychologically safe multi-rater feedback systems to evaluate employee performance. Traditional spreadsheet-based reviews fail to provide controlled lifecycle enforcement, role-based access governance, anonymous feedback protection, immutable audit trails, and aggregated analytics.
Solution:
A governance-first platform with strong anonymity enforcement, deterministic workflow control, and enterprise-grade audit logging:
• Structured feedback cycle lifecycle  (Draft → Nomination → Finalized → Active → Closed → Released → Archived)
• Multi-reviewer types: Self, Manager, Peer, Direct Reports
• Configurable anonymity states per reviewer type
• Peer threshold enforcement (minimum 3 submissions before peer data is visible)
• Snapshot-based reviewer assignment - org changes do not affect active cycles
• Immutable post-closure data storage enforced at DB trigger level
• Role-based dashboard analytics with stored aggregation
• Append-only audit logging for all sensitive actions
• Controlled async processing with intelligent reminder escalation engine
• Modular Monolith architecture - single deployable, clean internal module boundaries

2.2 Architecture Principles
Monolith First:
• Single deployable backend service - no distributed microservices
• Clear internal module boundaries enforced by folder structure and domain rules
• Can extract to microservices later if scale demands it
• All modules in one codebase with strict interface contracts
Database as Source of Truth:
• All workflow state stored in PostgreSQL
• ACID transactions for all state transitions - no partial state
• No external workflow engine (Temporal, Camunda, Step Functions) - complexity overkill
• DB-level ENUMs and triggers enforce type safety and immutability
Identity Separation:
• Feedback content stored separately from reviewer identity mapping
• Masking enforced at service layer - identity never appears in API response DTOs
• Only Security module can access identity_mappings table
Event-Driven Internal Processing:
• State transitions trigger background jobs (reminders, auto-close, aggregation)
• Idempotent job execution - safe to retry on any failure
• Cron-based schedulers, no external queue needed at this scale

2.3 Architecture Alternatives Considered
Decision: Modular Monolith
Alternatives Considered:
• Microservices - distributed system overhead not justified for 100-200 users
• Simple Monolith - no module boundaries, becomes unmaintainable as features grow
• Serverless (AWS Lambda) - cold start issues, complex state management for stateful workflows
Our System Characteristics:
• ✅  Single organization, predictable load pattern
• ✅  Linear state machine workflows - no distributed coordination needed
• ✅  Fast execution - state transitions in milliseconds, not hours
• ✅  Single team - monolith development speed advantage
• Architecture designed to be portable to AWS with no code changes
For 100-200 users, a Modular Monolith gives all the benefits of clean architecture without microservices overhead. We will revisit if scale grows beyond 1000 users.

3. WORKFLOW ARCHITECTURE
3.1 Cycle State Machine
The system implements a strict unidirectional cycle lifecycle. No backward transitions are permitted under normal operation. Reopen requires Super Admin override which is always audit logged with reason.



3.2 Workflow Implementation Strategy
Decision: Database-driven state machine  (ENUM status fields + service layer validation + DB triggers)
Alternatives Considered:
• Temporal - distributed workflow orchestration (for complex microservices, overkill for us)
• AWS Step Functions - serverless workflow service (requires AWS, adds cost and vendor lock-in)
• Camunda - Business Process Management platform (enterprise BPM, far too heavy)
• Apache Airflow - workflow scheduler (designed for data pipelines, not request-driven feedback)
Our Workflow Characteristics:
• ✅  Simple linear state chain  (DRAFT → NOMINATION → FINALIZED → ACTIVE → CLOSED → RELEASED → ARCHIVED)
• ✅  No complex branching logic or parallel execution paths
• ✅  Fast execution - transitions complete in milliseconds
• ✅  Deterministic transitions - clear rules, no ambiguity
• Single service (Modular Monolith) - no distributed coordination needed
For this scale and complexity, a database-driven state machine is the correct choice. We will revisit if workflow complexity increases significantly.

3.3 Reviewer Task State Machine
Each reviewer task follows its own lifecycle. LOCKED is a terminal state - no modification is permitted after the cycle closes. Reminders stop automatically when a reviewer reaches SUBMITTED.



3.4 Cycle State Transition Table

Current StateActionNext StateActorTransaction RequiredDRAFTActivate with nominationNOMINATIONHRYesDRAFTActivate - skip nominationFINALIZEDHRYesNOMINATIONDeadline reachedFINALIZEDSystemYesFINALIZEDAuto startACTIVESystemYesACTIVEDeadline reachedCLOSEDSystem (Scheduler)YesCLOSEDRelease resultsRESULTS_RELEASEDHRYesRESULTS_RELEASEDArchiveARCHIVEDHRYesAny (emergency)Super Admin overridePrevious stateSuper AdminYes + Audit Log
Key Points:
• No backward transitions allowed under any normal circumstance
• Every transition is wrapped in a database transaction - all-or-nothing
• All state changes are automatically audit logged
• Super Admin reopen creates an immutable audit record with reason

3.5 Snapshot Enforcement
At finalization, the current org hierarchy is snapshotted. Future org changes do NOT affect an active cycle.

Finalization Transaction (atomic - all or nothing):
──────────────────────────────────────────────────
1.  BEGIN TRANSACTION
2.  SELECT review_cycles WHERE id = :id FOR UPDATE
3.  Validate state = DRAFT or NOMINATION
4.  SNAPSHOT org_hierarchy  →  freeze into reviewer_tasks
5.  INSERT reviewer_tasks (all assignments, bulk)
6.  Lock config: peer_min, peer_max, anonymity, template_id
7.  UPDATE cycle state  →  FINALIZED
8.  COMMIT

If crash before COMMIT  →  automatic ROLLBACK, no partial tasks

3.6 Key Workflow Features
Peer Nomination:
• Minimum 3 peers required - non-configurable during active cycle
• Maximum peers configurable per cycle
• Self-nomination not permitted
• Nomination restricted to cycle participants only
• Manager override allowed (audit logged)
Audit Trail:
• Every state change logged in audit_logs table
• Records: actor, action, old/new state, timestamp, IP address, user agent
• Immutable - append-only, no UPDATE or DELETE permitted
• For compliance, debugging, and legal defensibility

4.2 Peer Threshold Rule
Minimum 3 peer submissions required before peer data appears in any report.
If peer_submissions < 3:
• Peer section is hidden from all reports and dashboard views
• Peer scores are excluded from aggregation computation
• Identity inference prevention maintained even from aggregate data
• Rule is non-overridable during an active cycle - even Super Admin cannot bypass mid-cycle

5.2 Stored Aggregation Model
Decision: Store aggregated results permanently at cycle closure (Option A)
Alternatives Considered:
• Dynamic recalculation on every dashboard load - slow, non-deterministic, dangerous if logic changes
• Cache layer (Redis) with aggregation - adds infrastructure complexity, still not legally defensible
Why Stored Aggregation was selected:
• ✅  Deterministic - results never change over time even if scoring logic is updated later
• ✅  Auditable - frozen results are legally defensible for HR performance decisions
• ✅  Fast dashboard - reads precomputed values, no heavy joins on every view
• Safe - a future code change cannot silently alter historical reports

Cycle Closure Processing Flow:
───────────────────────────────────────────────────────────
1.  Scheduler detects cycles past review_deadline
2.  SELECT review_cycles WHERE id = :id FOR UPDATE  (row lock)
3.  Check if aggregated_results already exist  →  exit if yes (idempotent)
4.  Lock all reviewer_tasks  →  LOCKED
5.  Apply peer threshold rule (exclude peer data if < 3 submissions)
6.  Compute per-reviewer-type averages
7.  Compute overall_score, variance, risk indicators
8.  INSERT aggregated_results  (UNIQUE per cycle + reviewee)
9.  UPDATE cycle state  →  CLOSED
10. COMMIT

5.4 Intelligent Reminder Engine
Decision: Intelligent escalation model (Option C) - no HR configuration needed in Phase 1

Day 1-2 after activationNo reminder  (settling period - reviewer needs time to plan)Day 3 to N-3Daily reminder to pending reviewersLast 3 days before deadlineDaily reminder continuesFinal dayMorning + afternoon  (2 reminders per day)After deadlineEscalate to manager + HR notificationAfter SUBMITTEDReminders stop immediately - excluded from next scheduler run

Reminder eligibility filter (live DB query every scheduler run):
  cycle_state = ACTIVE
  AND task_status IN (PENDING, IN_PROGRESS)
  AND review_deadline > NOW()

After submission:
  task_status = SUBMITTED  →  automatically excluded next run
  No stored queue - always a live filter, never stale state

6.1 Schema Overview
Core Entities:

  ↓ linked via
org_hierarchy (manager-employee tree)
  ↓ used to generate
review_cycles (lifecycle orchestrator)
  ↓ finalization creates

Supporting:
  identity_mappings  (Security module only)

Design Principles:
• Foreign keys with CASCADE/RESTRICT based on business logic - RESTRICT for career-impacting data
• PostgreSQL ENUM types for all state fields - DB-level type safety, not VARCHAR
• DB triggers enforce immutability as a second safety layer
• Normalized answers table (not JSONB) - enables question-level analytics and indexing
• Comprehensive indexes for all common query patterns

7. SECURITY ARCHITECTURE
7.1 Access Enforcement Model
All requests pass through a Centralized Access Guard Middleware. No controller directly checks roles - this prevents bypass via controller injection.
Every request validates (in order):
  1.  Authentication token validity
  2.  Role authorization (RBAC check)
  3.  Scope validation (cycle ownership)
  4.  Anonymity state enforcement
  5.  Threshold rule compliance

7.2 Immutability Enforcement - DB Triggers
Two-layer safety: service layer validates transitions, DB triggers prevent bypass. Even direct DB access or accidental migrations cannot corrupt core data.

-- Prevent ARCHIVED cycle modification
CREATE OR REPLACE FUNCTION validate_cycle_transition()
RETURNS TRIGGER AS $$
BEGIN
  IF OLD.state = 'ARCHIVED' THEN
    RAISE EXCEPTION 'Archived cycles cannot be modified';
  END IF;
  RETURN NEW;
END; $$ LANGUAGE plpgsql;

CREATE TRIGGER cycle_state_guard
  BEFORE UPDATE ON review_cycles FOR EACH ROW
  WHEN (OLD.state <> NEW.state)
  EXECUTE FUNCTION validate_cycle_transition();

-- Prevent LOCKED task modification
CREATE OR REPLACE FUNCTION prevent_locked_task_update()
RETURNS TRIGGER AS $$
BEGIN
  IF OLD.status = 'LOCKED' THEN
    RAISE EXCEPTION 'Locked reviewer tasks cannot be modified';
  END IF;
  RETURN NEW;
END; $$ LANGUAGE plpgsql;

-- Feedback + aggregation immutability
CREATE OR REPLACE FUNCTION prevent_immutable_update()
RETURNS TRIGGER AS $$
BEGIN RAISE EXCEPTION 'This record is immutable';
END; $$ LANGUAGE plpgsql;

-- Delete prevention on all core tables
CREATE OR REPLACE FUNCTION prevent_core_delete()
RETURNS TRIGGER AS $$
BEGIN RAISE EXCEPTION 'Deletion not allowed on core tables';
END; $$ LANGUAGE plpgsql;

8. CONCURRENCY & FAILURE RECOVERY
8.1 Concurrency Strategy
All critical state transitions use SELECT ... FOR UPDATE (row-level locking). This prevents race conditions in finalization, closure, and aggregation.

-- Cycle finalization
BEGIN;
  SELECT * FROM review_cycles WHERE id = :cycle_id FOR UPDATE;
  -- Validate state, snapshot org, INSERT tasks, UPDATE state
COMMIT;

-- Idempotent aggregation job
SELECT 1 FROM aggregated_results WHERE cycle_id = :cycle_id LIMIT 1;
-- If row exists  →  EXIT (job already completed, safe to skip)

-- Duplicate submission protection
-- UNIQUE (reviewer_task_id) on feedback_responses
-- UNIQUE (response_id, question_id) on feedback_answers
-- Double-click submit fails safely on second attempt

8.2 Failure Recovery

Failure ScenarioSystem BehaviorRecoveryCrash during finalizationTransaction rolls back automaticallyNo partial tasks - re-trigger finalizationCrash during aggregation (before insert)No aggregated_results row existsJob retried - idempotency check runs firstCrash after CLOSED, before aggregationCycle = CLOSED, results missingScheduler detects and re-triggers aggregationDouble scheduler executionSecond job finds results exist - exitsNo duplicate computation, no data corruptionDB connection failure mid-transactionROLLBACK on timeoutOperation can be safely retried
